Securite

Deux classes : le hachage des mots de passe et la session de l'utilisateur connecte.

GestionnaireMotDePasse

Fichier : src/main/java/com/magasin/securite/GestionnaireMotDePasse.java

public final class GestionnaireMotDePasse {

    private static final int COUT = 12;

    private GestionnaireMotDePasse() {}

    public static String hacher(String motDePasseClair) {
        if (motDePasseClair == null || motDePasseClair.isEmpty()) {
            throw new IllegalArgumentException("Mot de passe vide");
        }
        return BCrypt.hashpw(motDePasseClair, BCrypt.gensalt(COUT));
    }

    public static boolean verifier(String motDePasseClair, String hashStocke) {
        if (motDePasseClair == null || hashStocke == null || hashStocke.isEmpty()) {
            return false;
        }
        try {
            return BCrypt.checkpw(motDePasseClair, hashStocke);
        } catch (IllegalArgumentException e) {
            return false;
        }
    }
}

Pourquoi BCrypt ?

BCrypt est un algorithme de hachage concu pour les mots de passe. Trois proprietes le distinguent d'un MD5 ou d'un SHA-256 simple :

  • Sel automatique : BCrypt.gensalt genere un sel aleatoire qui est integre au hash final. Deux mots de passe identiques donnent donc deux hashes differents. Inutile de stocker le sel a part.
  • Cout configurable : le parametre 12 (notre constante COUT) signifie 212 = 4096 iterations. On peut augmenter ce nombre quand les ordinateurs deviennent plus rapides.
  • Resistant aux tables arc-en-ciel : sans sel, un attaquant precalcule des hashes pour des millions de mots de passe courants. Le sel rend ces tables inutiles.

Format du hash

Un hash BCrypt ressemble a : $2a$12$N9qo8uLOickgx2ZMRZoMye.... Lecture :

  • $2a : version de l'algorithme BCrypt.
  • $12 : cout (12 dans notre cas).
  • $N9qo8uLOickgx2ZMRZoMye : sel encode en base64 (22 caracteres).
  • Le reste : le hash effectif.

La longueur totale est 60 caracteres. C'est pour ca que la colonne password de la table utilisateur est dimensionnee a 255 : on a de la marge.

Verification

BCrypt.checkpw(clair, hash) extrait le cout et le sel du hash stocke, re-hache le mot de passe en clair avec ces parametres, puis compare avec le hash. Si les deux sont egaux, le mot de passe est bon.

Le constructeur prive

Toutes les methodes sont static. On rend le constructeur prive pour empecher l'instanciation accidentelle de la classe.

SessionUtilisateur

Fichier : src/main/java/com/magasin/securite/SessionUtilisateur.java

public final class SessionUtilisateur {

    private static Utilisateur utilisateurCourant;

    private SessionUtilisateur() {}

    public static void connecter(Utilisateur u) {
        utilisateurCourant = u;
    }

    public static void deconnecter() {
        utilisateurCourant = null;
    }

    public static Utilisateur utilisateurCourant() {
        return utilisateurCourant;
    }

    public static boolean estConnecte() {
        return utilisateurCourant != null;
    }

    public static boolean estAdministrateur() {
        return utilisateurCourant != null
            && utilisateurCourant.estAdministrateur();
    }
}

Pourquoi un singleton statique ?

L'application n'a qu'un seul utilisateur connecte a la fois. Plutot que de passer cet utilisateur en parametre a chaque fenetre, on le stocke dans une variable statique accessible globalement. FenetrePrincipale et PanneauUtilisateurs appellent SessionUtilisateur.estAdministrateur() pour decider quoi afficher.

Limites

Pour une vraie application multi-fenetre, on prefererait un objet injecte explicitement. Le singleton statique a l'avantage d'etre simple a comprendre pour la soutenance.

A retenir : aucun mot de passe en clair n'est jamais stocke ni transmis apres l'authentification. Le hash arrive depuis Appwrite, est passe a BCrypt avec le mot de passe en clair, puis BCrypt repond oui ou non.
