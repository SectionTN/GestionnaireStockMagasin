Controleurs

Les controleurs sont des intermediaires fins entre les vues et les services. Pour la plupart, ils delegent directement. Le controleur de connexion fait un peu plus.

ControleurConnexion

Fichier : src/main/java/com/magasin/controleur/ControleurConnexion.java

public static void connecter(FenetreConnexion fenetre,
                              String username,
                              String motDePasse,
                              Consumer<String> auxErreurs) {
    if (username == null || username.isBlank()) {
        auxErreurs.accept("Veuillez saisir un nom d'utilisateur.");
        return;
    }
    if (motDePasse == null || motDePasse.isEmpty()) {
        auxErreurs.accept("Veuillez saisir un mot de passe.");
        return;
    }

    new SwingWorker<Utilisateur, Void>() {
        protected Utilisateur doInBackground() throws Exception {
            return ServiceUtilisateur.authentifier(username, motDePasse);
        }
        protected void done() {
            try {
                Utilisateur u = get();
                if (u == null) {
                    auxErreurs.accept("Identifiants incorrects.");
                    return;
                }
                SessionUtilisateur.connecter(u);
                SwingUtilities.invokeLater(() -> {
                    fenetre.dispose();
                    new FenetrePrincipale().setVisible(true);
                });
            } catch (Exception ex) {
                Throwable cause = ex.getCause() != null ? ex.getCause() : ex;
                auxErreurs.accept("Echec : " + cause.getMessage());
            }
        }
    }.execute();
}

Logique pas a pas

  1. Validation cote client : on refuse les champs vides avant tout appel reseau.
  2. Appel asynchrone : SwingWorker execute authentifier sur un thread de fond. Pendant ce temps, l'interface reste reactive.
  3. Resultat : trois cas possibles dans done() :
    • u == null : pas d'utilisateur trouve ou mot de passe faux. On appelle auxErreurs avec "Identifiants incorrects".
    • u != null : succes. On stocke dans la session, on ferme l'ecran de connexion et on ouvre le tableau de bord.
    • Exception : probleme reseau ou Appwrite. On extrait la cause et on l'affiche.

Pourquoi un Consumer<String> pour les erreurs ?

Le controleur ne sait rien des composants Swing. Il ne peut pas appeler directement etiquetteMessage.setText. La vue lui passe une fonction (lambda) : "voici ce qu'il faut faire avec un message d'erreur". Cette inversion de dependance garde le controleur testable.

ControleurProduits

Fichier : src/main/java/com/magasin/controleur/ControleurProduits.java

public final class ControleurProduits {

    private ControleurProduits() {}

    public static List<Produit> listerTout() throws AppwriteException {
        return ServiceProduit.listerTout();
    }
    public static Produit ajouter(Produit p) throws AppwriteException {
        return ServiceProduit.ajouter(p);
    }
    public static Produit modifier(Produit p) throws AppwriteException {
        return ServiceProduit.modifier(p);
    }
    public static void supprimer(String id) throws AppwriteException {
        ServiceProduit.supprimer(id);
    }
}

Ce controleur n'ajoute rien de visible. Mais il rend la couche d'abstraction explicite : si demain on change de backend (par exemple un cache local), on modifie le controleur sans toucher aux panneaux.

ControleurFournisseurs et ControleurCommandes

Memes patrons. Le controleur des commandes expose en plus listerProduits() pour alimenter le combo du dialogue de commande.

public static List<Produit> listerProduits() throws AppwriteException {
    return ServiceProduit.listerTout();
}
Question typique : "Pourquoi des controleurs si fins ?" Reponse : pour respecter la separation des couches du MVC. Meme s'ils sont triviaux aujourd'hui, ils centralisent l'orchestration. Quand on devra ajouter une regle metier (par exemple "interdire la suppression d'un produit avec des commandes liees"), on aura un endroit naturel ou la mettre.
