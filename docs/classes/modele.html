Modeles (POJO)

Quatre classes simples qui representent les entites manipulees par l'application. Chaque champ correspond a une colonne d'une table Appwrite. L'id est une chaine car Appwrite genere un identifiant alphanumerique (le $id), pas un entier auto-incremente.

Produit

Fichier : src/main/java/com/magasin/modele/Produit.java

public class Produit {
    private String id;
    private String nom;
    private int quantite;
    private double prix;
    private String fournisseur;

    public Produit() {}

    public Produit(String id, String nom, int quantite, double prix, String fournisseur) {
        this.id = id;
        this.nom = nom;
        this.quantite = quantite;
        this.prix = prix;
        this.fournisseur = fournisseur;
    }

    // getters et setters ...

    @Override
    public String toString() {
        return nom + " (qte=" + quantite + ", prix=" + prix + ")";
    }
}

Detail des champs

ChampTypeRole
idStringIdentifiant de la ligne dans Appwrite. Null tant que le produit n'a pas ete sauvegarde.
nomStringLibelle du produit affiche a l'utilisateur.
quantiteintStock disponible. Modifie automatiquement lors d'une commande.
prixdoublePrix unitaire. double a ete choisi pour la simplicite (un vrai projet utiliserait BigDecimal).
fournisseurStringNom du fournisseur. C'est volontairement une chaine et pas une cle etrangere : le sujet le specifie ainsi.

Pourquoi un toString() personnalise ?

Dans le dialogue de commande, on utilise un JComboBox<Produit>. Le combo affiche pour chaque element ce que renvoie toString(). On profite donc de la methode pour montrer le nom, la quantite en stock et le prix.

La classe la plus simple. Le toString() retourne juste le nom : pratique si on veut un jour ajouter un combo Fournisseur quelque part.

Commande

Fichier : src/main/java/com/magasin/modele/Commande.java

public class Commande {
    private String id;
    private String produitId;       // reference vers Produit.id
    private int quantite;
    private String dateCommande;    // ISO-8601, ex: 2026-05-21

    public Commande() {}
    public Commande(String id, String produitId, int quantite, String dateCommande) { /* ... */ }
    // getters et setters
}

Detail

  • produitId : reference au produit commande. Stocke la valeur de Produit.id. La table Appwrite utilise le nom de colonne produit_id (snake_case cote serveur, camelCase cote Java) ; la conversion se fait dans le service.
  • dateCommande : stockee sous forme de chaine ISO-8601 (par exemple 2026-05-21). Appwrite a un type DateTime, mais une chaine simplifie la serialisation et reste lisible.

Utilisateur

Fichier : src/main/java/com/magasin/modele/Utilisateur.java

public class Utilisateur {
    public static final String ROLE_ADMINISTRATEUR = "administrateur";
    public static final String ROLE_GESTIONNAIRE   = "gestionnaire";

    private String id;
    private String username;
    private String password;   // hash bcrypt, jamais en clair
    private String role;

    public Utilisateur() {}
    public Utilisateur(String id, String username, String password, String role) { /* ... */ }
    // getters et setters

    public boolean estAdministrateur() {
        return ROLE_ADMINISTRATEUR.equalsIgnoreCase(role);
    }
}

Constantes de role

Les valeurs de role sont dupliquees nulle part dans le code : on passe par les constantes ROLE_ADMINISTRATEUR et ROLE_GESTIONNAIRE. Si le nom change un jour, on ne touche qu'a un seul fichier.

Methode estAdministrateur()

Plutot que de comparer la chaine partout ("administrateur".equals(u.getRole())), on a ajoute cette methode qui :

  • Compare sans tenir compte de la casse (equalsIgnoreCase) pour eviter les bugs si quelqu'un saisit "Administrateur" dans la base.
  • Encapsule la logique : si demain on ajoute un role super_admin, on modifie juste cette methode.
A retenir pour la soutenance : les POJO ne contiennent aucune logique reseau ou interface. Ce sont des structures de donnees pures. Toute la logique vit dans les services et les controleurs.
