Questions du prof

Liste de questions probables avec des reponses preparees. L'idee : pouvoir reciter le raisonnement, pas juste connaitre le code par coeur.

Architecture et conception

Q. Quel design pattern utilisez-vous ?
Le projet suit l'esprit du MVC (Modele-Vue-Controleur). Le modele, ce sont les POJO dans modele/. La vue, c'est Swing dans vue/. Le controleur fait le pont. La couche service (service/) ajoute une separation supplementaire : elle isole les appels Appwrite pour qu'on puisse changer de backend sans toucher au reste.
Q. Pourquoi separer service et controleur ?
Le controleur connait l'interface (il appelle SwingWorker, il dispatche les messages d'erreur). Le service connait la base de donnees (il construit le JSON, il fait les appels HTTP). Si on les melangeait, on aurait du code Swing dans la logique reseau, ou inversement. C'est ce qu'on appelle la separation des responsabilites.
Q. Pourquoi des classes final avec constructeur prive ?
Pour les classes qui ne contiennent que des methodes statiques (services, utilitaires, securite). On veut empecher l'instanciation : new ServiceProduit() n'aurait aucun sens. Le constructeur prive bloque l'appel, et final empeche l'heritage qui contournerait la regle.

Base de donnees et Appwrite

Q. Pourquoi Appwrite et pas MySQL avec JDBC ?
Pour apprendre a integrer un service distant via REST. JDBC, on aura l'occasion d'en faire ailleurs. Avec Appwrite on doit construire un client HTTP, gerer les en-tetes d'authentification, parser le JSON. C'est plus proche d'un cas reel d'integration d'API.
Q. Pourquoi l'id est-il une chaine et pas un int ?
Appwrite genere des identifiants alphanumeriques (par exemple 672a4f8b00123abc), pas des entiers auto-incrementes comme MySQL. C'est une chaine cote serveur, donc on garde une chaine cote Java. Ca evite des conversions inutiles. Pour les jointures (la commande qui reference un produit), on stocke l'id du produit comme une chaine dans la commande.
Q. Comment vous gerez la cle etrangere produit_id ?
C'est un champ chaine dans la table commande. On y stocke la valeur de Produit.$id. Aucune contrainte d'integrite cote serveur (Appwrite ne fait pas de FK strictes). C'est l'application qui verifie que le produit existe avant de creer la commande, via ServiceProduit.obtenirParId.
Q. Que se passe-t-il si on supprime un produit qui a des commandes ?
Aujourd'hui, rien ne l'empeche. Les commandes referenceront un produit qui n'existe plus, et le tableau affichera "(produit inconnu)" grace au fallback dans PanneauCommandes. Une amelioration serait de verifier dans ServiceProduit.supprimer qu'aucune commande ne reference ce produit.

Securite

Q. Pourquoi BCrypt ?
BCrypt est concu pour les mots de passe, contrairement a MD5 ou SHA-256 qui sont rapides (et donc faciles a casser par force brute). BCrypt integre un sel aleatoire dans chaque hash (deux mots de passe identiques donnent des hashes differents) et un cout configurable (12 dans notre code, donc 4096 iterations). Si les ordinateurs deviennent plus rapides, on augmente le cout.
Q. Qu'est-ce qu'un sel ?
Une valeur aleatoire ajoutee au mot de passe avant le hachage. Sans sel, un attaquant peut precalculer des hashes pour des millions de mots de passe courants (table arc-en-ciel). Avec un sel different par utilisateur, ces tables deviennent inutiles : il faudrait recalculer une table par sel. BCrypt genere le sel automatiquement et le stocke dans le hash final.
Q. Pourquoi ne pas distinguer "utilisateur inexistant" de "mauvais mot de passe" ?
Pour ne pas aider un attaquant. S'il essaie un nom d'utilisateur et qu'on lui repond "utilisateur inexistant", il sait qu'il peut tester d'autres noms. Si on repond toujours "identifiants incorrects", il ne sait pas si le nom existe.
Q. La cle API est dans config.properties, c'est sur ?
Le fichier est dans .gitignore donc il ne va pas sur Git. Pour une vraie application, on prefererait une variable d'environnement ou un vault. Pour un projet d'examen, c'est acceptable. La cle a des scopes limites a tables.* et rows.* ; meme compromise, l'attaquant ne peut pas changer la configuration du projet Appwrite.
Q. Comment vous gerez les roles ?
Deux roles dans la base : administrateur et gestionnaire. Apres connexion, l'utilisateur est stocke dans SessionUtilisateur. La FenetrePrincipale teste SessionUtilisateur.estAdministrateur() pour decider si le panneau Utilisateurs est ajoute. Un gestionnaire de stock n'a meme pas l'onglet.

Swing et interface

Q. Pourquoi Swing et pas JavaFX ?
Le sujet du projet demandait Swing. On l'a habille avec FlatLaf (un look-and-feel moderne, theme IntelliJ) pour eviter l'apparence Java par defaut.
Q. C'est quoi le EDT ?
L'Event Dispatch Thread, le thread unique sur lequel Swing dessine et reagit aux clics. Toute creation ou modification de composant doit se faire dessus. Si on lance un appel reseau dans actionPerformed, l'interface fige tant que la reponse n'arrive pas. C'est pour ca qu'on utilise SwingWorker partout.
Q. Pourquoi SwingWorker et pas un Thread simple ?
SwingWorker donne deux callbacks : doInBackground sur un thread de fond et done sur l'EDT. Pas besoin de gerer manuellement le passage entre les deux. Avec un Thread brut, on devrait appeler SwingUtilities.invokeLater a la main pour mettre a jour l'interface.
Q. Comment marche le CardLayout ?
Un layout qui empile plusieurs composants au meme endroit et n'en montre qu'un seul a la fois. On nomme chaque composant a l'ajout (zoneContenu.add(panneau, "CLE")), puis on appelle cardLayout.show(zoneContenu, "CLE") pour basculer. Les boutons de la barre laterale appellent cette methode.

Code et bonnes pratiques

Q. Pourquoi avoir mis le code en francais ?
Le projet est un projet d'examen, evalue en francais. Les variables, methodes, classes et messages d'erreur sont en francais pour cette raison. Dans un contexte professionnel international, on coderait plutot en anglais.
Q. Comment vous gerez les erreurs ?
AppwriteException est levee par la couche service en cas de probleme. Elle traverse les controleurs et est attrapee dans la vue, qui appelle GestionnaireErreurs.erreur. Le dialogue affiche le message d'origine, par exemple "Stock insuffisant pour 'Clavier'. Disponible : 5, demande : 10."
Q. Pourquoi utiliser Gson plutot que de parser le JSON a la main ?
Gson est une bibliotheque Google maintenue depuis 2008. Parser du JSON a la main est faisable mais ennuyeux et source d'erreurs (gestion des chaines echappees, des valeurs nulles, etc). Gson fait 250 ko et resout tous ces problemes.
Q. C'est quoi java.net.http.HttpClient ?
Le client HTTP officiel ajoute dans Java 11. Il remplace l'ancien HttpURLConnection qui etait verbeux. Notre client envoie une HttpRequest et recoit une HttpResponse. On peut faire du synchrone (send) ou de l'asynchrone (sendAsync). On utilise du synchrone parce qu'on est deja dans un SwingWorker.

Specifiques commandes

Q. Que se passe-t-il si on commande plus que le stock disponible ?
ServiceCommande.ajouter recupere le produit, compare la quantite demandee au stock, et leve une AppwriteException avec un message clair si le stock est insuffisant. La vue affiche le message dans un JOptionPane. Aucune ecriture n'a lieu dans la base.
Q. Et si l'utilisateur change le produit d'une commande existante ?
modifier compare l'ancien et le nouveau produit. Si c'est le meme, on calcule la difference de quantite et on l'applique au stock. Si c'est un produit different, on rembourse l'ancien (ajout au stock) puis on debite le nouveau (soustraction). La verification du stock du nouveau produit se fait avant de toucher quoi que ce soit, pour ne pas laisser la base dans un etat incoherent en cas d'echec.
Q. La modification du stock et l'ecriture de la commande sont-elles atomiques ?
Non. Appwrite ne supporte pas les transactions multi-collections en REST. Si l'une des deux operations echoue apres l'autre, l'etat peut etre incoherent. Pour une vraie application, on utiliserait une fonction serveur Appwrite ou une vraie base SQL avec transaction. Pour ce projet, on documente la limite et on choisit l'ordre qui laisse le moins de degats : modifier le produit avant de creer la commande.

Maven et build

Q. C'est quoi le shade plugin ?
Un plugin Maven qui regroupe l'application et toutes ses dependances dans un seul JAR executable (un "fat jar"). Sans lui, il faudrait lancer l'app avec un classpath complet qui pointe vers FlatLaf, Gson, jBCrypt. Avec lui, java -jar target/gestionnaire-stock-magasin.jar suffit.
Q. Pourquoi ${exec.mainClass} dans le pom ?
Pour pouvoir lancer une autre classe que l'app principale via -Dexec.mainClass=.... Concretement, ca nous sert a executer GenererHashAdmin sans modifier le pom. Si on codait com.magasin.Application en dur dans la config du plugin, l'argument en ligne de commande serait ignore.

Limites et ameliorations

Q. Qu'est-ce qu'on ameliorerait ?
  • Transactions reelles pour la commande (via une fonction serveur Appwrite).
  • Pagination des listes (si on a beaucoup de produits, le listerTout devient lent).
  • Cache local pour reduire les allers-retours reseau.
  • Recherche / filtre dans les tableaux.
  • Export CSV des commandes.
  • Tests unitaires (JUnit) pour les services et la securite.
Q. Avez-vous teste l'application ?
On a teste manuellement chaque parcours : connexion admin, connexion gestionnaire, CRUD complet sur les quatre tables, verification que le panneau Utilisateurs n'apparait pas pour un gestionnaire, verification du blocage de commande quand le stock est insuffisant. Pas de tests automatises (JUnit) pour ce projet : c'est une amelioration prevue.
