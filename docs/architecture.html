Architecture

L'application est organisee en couches, dans l'esprit MVC. Chaque couche a une responsabilite unique et ne connait que la couche en dessous.

Schema des couches

+-------------------------------------------------------+
|  Vue (Swing)  : Fenetres, panneaux, dialogues         |
+-------------------------------------------------------+
                       v
+-------------------------------------------------------+
|  Controleur   : Validation, SwingWorker, orchestration|
+-------------------------------------------------------+
                       v
+-------------------------------------------------------+
|  Service      : Logique metier + appels Appwrite      |
+-------------------------------------------------------+
                       v
+-------------------------------------------------------+
|  ClientAppwrite : HTTP + JSON (java.net.http + Gson)  |
+-------------------------------------------------------+
                       v
                  Appwrite Cloud

Flux de connexion

  1. L'utilisateur ouvre l'app. Application.main() initialise FlatLaf, lit la configuration puis affiche FenetreConnexion.
  2. L'utilisateur saisit son login et son mot de passe puis valide.
  3. ControleurConnexion recoit l'evenement et lance un SwingWorker pour ne pas geler l'interface.
  4. Le worker appelle ServiceUtilisateur.authentifier(user, mdp). Ce service liste les utilisateurs et compare le mot de passe avec le hash via GestionnaireMotDePasse.verifier.
  5. Si OK, on stocke l'utilisateur dans SessionUtilisateur et on ouvre FenetrePrincipale. Sinon on affiche le message d'erreur sous le formulaire.

Flux d'un CRUD typique

Exemple : ajout d'un produit.

  1. L'utilisateur clique sur "Ajouter" dans PanneauProduits.
  2. Le panneau ouvre DialogueProduit. Le formulaire valide les entrees (nom non vide, quantite positive, prix positif).
  3. A la validation, le panneau lance un SwingWorker qui appelle ControleurProduits.ajouter(p).
  4. Le controleur delegue a ServiceProduit.ajouter(p), qui construit le JSON et appelle ClientAppwrite.creerLigne.
  5. Le client envoie un POST /v1/tablesdb/{db}/tables/produit/rows avec les en-tetes Appwrite.
  6. Au retour, le panneau recharge la table et affiche un message de succes.

Gestion des erreurs reseau

Toute reponse HTTP hors 2xx leve une AppwriteException qui contient le code et le message renvoye par Appwrite. Cette exception traverse les couches jusqu'a la vue, qui l'affiche via GestionnaireErreurs.erreur.

Concurrence

Swing tourne sur un thread unique : l'EDT (Event Dispatch Thread). Bloquer ce thread pour un appel HTTP gele l'interface. Toutes les operations reseau passent donc par SwingWorker :

new SwingWorker<Produit, Void>() {
    protected Produit doInBackground() throws Exception {
        return ServiceProduit.ajouter(p);    // hors EDT
    }
    protected void done() {
        get();                                // de retour sur l'EDT
        rafraichir();
    }
}.execute();
