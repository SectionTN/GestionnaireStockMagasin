Walkthrough du code

Lecture ligne par ligne des passages cles : le client HTTP Appwrite, l'authentification, la gestion du stock lors d'une commande. On a privilegie les parties qui ne sont pas immediatement evidentes.

1. Un appel REST de bout en bout

Exemple : ajouter un produit. Suivons l'appel depuis le bouton jusqu'au serveur.

Etape 1 — La vue declenche

// PanneauProduits.java
private void ajouter() {
    Produit p = DialogueProduit.afficher(this, null);
    if (p == null) return;
    executerService(() -> ControleurProduits.ajouter(p), "Produit ajoute.");
}

DialogueProduit.afficher ouvre le formulaire et retourne un Produit rempli, ou null si annulation. La lambda () -> ControleurProduits.ajouter(p) sera executee dans un SwingWorker.

Etape 2 — Le controleur delegue

// ControleurProduits.java
public static Produit ajouter(Produit p) throws AppwriteException {
    return ServiceProduit.ajouter(p);
}

Pas de traitement particulier ici. Le controleur se contente d'appeler le service. Si on devait verifier des regles metier (nom unique, prix dans une fourchette), c'est ici qu'on le ferait.

Etape 3 — Le service convertit en JSON

// ServiceProduit.java
public static Produit ajouter(Produit p) throws AppwriteException {
    JsonObject d = versJson(p);                   // POJO -> JSON
    JsonObject rep = ClientAppwrite.creerLigne(table(), d);  // appel reseau
    return depuisJson(rep);                       // JSON -> POJO
}

private static JsonObject versJson(Produit p) {
    JsonObject o = new JsonObject();
    o.addProperty("nom", p.getNom());
    o.addProperty("quantite", p.getQuantite());
    o.addProperty("prix", p.getPrix());
    o.addProperty("fournisseur", p.getFournisseur() == null ? "" : p.getFournisseur());
    return o;
}

Trois lignes. Conversion, appel, conversion inverse. La methode table() retourne "produit" via Configuration.tableProduit().

Etape 4 — Le client construit la requete HTTP

// ClientAppwrite.java
public static JsonObject creerLigne(String tableId, JsonObject donnees) {
    JsonObject corps = new JsonObject();
    corps.addProperty("rowId", "unique()");      // Appwrite genere l'id
    corps.add("data", donnees);

    HttpRequest req = requeteBase(basePourTable(tableId))
            .POST(BodyPublishers.ofString(corps.toString(), StandardCharsets.UTF_8))
            .build();
    return envoyer(req);
}

Le corps de la requete ressemble a :

{
  "rowId": "unique()",
  "data": {
    "nom": "Clavier mecanique",
    "quantite": 25,
    "prix": 89.99,
    "fournisseur": "TechPro"
  }
}

Etape 5 — Envoi et lecture de la reponse

private static JsonObject envoyer(HttpRequest req) throws AppwriteException {
    try {
        HttpResponse<String> rep = HTTP.send(req,
                BodyHandlers.ofString(StandardCharsets.UTF_8));
        int code = rep.statusCode();
        if (code >= 200 && code < 300) {
            return JsonParser.parseString(rep.body()).getAsJsonObject();
        }
        // sinon : on construit un message d'erreur lisible
        String msg = "Erreur HTTP " + code;
        try {
            JsonObject err = JsonParser.parseString(rep.body()).getAsJsonObject();
            if (err.has("message")) msg += " — " + err.get("message").getAsString();
        } catch (Exception ignore) {}
        throw new AppwriteException(msg, code);
    } catch (Exception e) {
        throw new AppwriteException("Echec reseau : " + e.getMessage(), -1);
    }
}

Appwrite repond avec le row cree :

{
  "$id": "672a4f8b00123abc",
  "nom": "Clavier mecanique",
  "quantite": 25,
  "prix": 89.99,
  "fournisseur": "TechPro",
  "$createdAt": "2026-05-21T14:30:00.000+00:00"
}

La methode depuisJson du service extrait $id via extraireId et reconstruit un Produit.

2. Authentification : le flot complet

Generer le hash a la creation

// ServiceUtilisateur.java
public static Utilisateur ajouter(String username, String motDePasseClair, String role) {
    JsonObject d = new JsonObject();
    d.addProperty("username", username);
    d.addProperty("password", GestionnaireMotDePasse.hacher(motDePasseClair));
    d.addProperty("role", role);
    return depuisJson(ClientAppwrite.creerLigne(table(), d));
}

Le mot de passe en clair est haché immediatement. Il n'est jamais ecrit en base, ni dans les logs.

Verifier au login

public static Utilisateur authentifier(String username, String motDePasseClair) {
    Utilisateur u = trouverParUsername(username);
    if (u == null) return null;
    if (!GestionnaireMotDePasse.verifier(motDePasseClair, u.getPassword())) return null;
    return u;
}

Ordre des verifications :

  1. Chercher l'utilisateur par nom. S'il n'existe pas, on renvoie null.
  2. Comparer le mot de passe en clair au hash stocke via BCrypt. checkpw re-hache le clair avec le sel extrait du hash et compare.
  3. Si l'un des deux echoue, on renvoie null. La vue affiche "Identifiants incorrects" sans distinguer les deux cas (pour ne pas dire a un attaquant que l'utilisateur existe).

Stocker dans la session

// ControleurConnexion.java
SessionUtilisateur.connecter(u);
SwingUtilities.invokeLater(() -> {
    fenetre.dispose();
    new FenetrePrincipale().setVisible(true);
});

SessionUtilisateur garde l'utilisateur en variable statique. FenetrePrincipale lira SessionUtilisateur.estAdministrateur() pour decider d'afficher ou non le panneau Utilisateurs.

3. Stock et commandes : la partie metier

Verification avant ajout

// ServiceCommande.java
public static Commande ajouter(Commande c) throws AppwriteException {
    Produit p = ServiceProduit.obtenirParId(c.getProduitId());
    if (p == null) {
        throw new AppwriteException("Produit introuvable.", -1);
    }
    if (c.getQuantite() > p.getQuantite()) {
        throw new AppwriteException(
            "Stock insuffisant pour \"" + p.getNom() + "\". "
            + "Disponible : " + p.getQuantite()
            + ", demande : " + c.getQuantite() + ".", -1);
    }
    p.setQuantite(p.getQuantite() - c.getQuantite());
    ServiceProduit.modifier(p);
    JsonObject rep = ClientAppwrite.creerLigne(table(), versJson(c));
    return depuisJson(rep);
}

Lecture ligne par ligne :

  • ServiceProduit.obtenirParId : GET /v1/tablesdb/{db}/tables/produit/rows/{id}. Recupere le produit actuel.
  • if (c.getQuantite() > p.getQuantite()) : la verification de stock. On compare la quantite demandee au stock disponible.
  • throw new AppwriteException(...) : on leve une exception verifiee avec un message en francais. La vue va l'attraper et l'afficher dans un JOptionPane.
  • p.setQuantite(p.getQuantite() - c.getQuantite()) : on debite le stock.
  • ServiceProduit.modifier(p) : PATCH sur la ligne produit.
  • ClientAppwrite.creerLigne : POST pour creer la commande.

Pourquoi modifier le produit AVANT la commande ?

Si on creait d'abord la commande puis on modifiait le produit, et que la modification echouait, on se retrouverait avec une commande sans debit du stock. L'inverse est aussi imparfait (modification reussie mais commande qui echoue laisse un stock decremente sans commande), mais c'est plus simple a detecter : un produit avec un stock errone est moins critique qu'une commande "fantome".

Une vraie solution serait une transaction. Appwrite ne supporte pas les transactions multi-collections en REST. Pour un projet d'examen, on documente la limite et on choisit le moindre mal.

Modification avec changement de produit

// Cas : l'utilisateur change le produit d'une commande existante
Produit ancien  = ServiceProduit.obtenirParId(ancienne.getProduitId());
Produit nouveau = ServiceProduit.obtenirParId(nouvelle.getProduitId());

if (nouveau.getQuantite() < nouvelle.getQuantite()) {
    throw new AppwriteException("Stock insuffisant pour \"" + nouveau.getNom() + "\". ...", -1);
}
ancien.setQuantite(ancien.getQuantite() + ancienne.getQuantite());   // rembourse
ServiceProduit.modifier(ancien);
nouveau.setQuantite(nouveau.getQuantite() - nouvelle.getQuantite()); // debite
ServiceProduit.modifier(nouveau);

Sequence importante : on verifie avant de toucher quoi que ce soit. Si on remboursait l'ancien produit puis on tentait de debiter le nouveau et que ca echouait, l'ancien serait deja credite et l'etat serait incoherent.

4. SwingWorker : pourquoi et comment

Swing tourne sur un seul thread (l'EDT). Toute operation longue sur ce thread fige la fenetre. Un appel HTTP peut prendre une seconde ou deux. Sans SwingWorker, l'utilisateur verrait l'app gelee.

new SwingWorker<List<Produit>, Void>() {
    protected List<Produit> doInBackground() throws Exception {
        return ControleurProduits.listerTout();      // hors EDT
    }
    protected void done() {                          // de retour sur l'EDT
        modele.setRowCount(0);
        for (Produit p : get()) {
            modele.addRow(new Object[]{p.getId(), p.getNom(), p.getQuantite(),
                                        p.getPrix(), p.getFournisseur()});
        }
    }
}.execute();

Deux methodes a implementer :

  • doInBackground() : execute sur un thread de fond. C'est ici que l'appel reseau a lieu.
  • done() : execute sur l'EDT apres doInBackground. C'est le seul endroit ou on touche aux composants Swing.

get() dans done() retourne le resultat de doInBackground, ou releve l'exception si elle s'est produite. C'est pour ca qu'on l'entoure d'un try/catch.

5. Le bricolage de la mise a l'echelle HiDPI

Sur un ecran haute densite, Swing affiche par defaut des composants minuscules. Le truc consiste a positionner sun.java2d.uiScale AVANT de toucher au moindre composant AWT.

public static void main(String[] args) {
    System.setProperty("sun.java2d.uiScale.enabled", "true");
    if (System.getProperty("sun.java2d.uiScale") == null) {
        System.setProperty("sun.java2d.uiScale",
            String.valueOf(detecterEchelle()));   // 2.0 sur Linux par defaut
    }
    // ...
}

private static double detecterEchelle() {
    String env = System.getenv("MAGASIN_UI_SCALE");
    if (env != null && !env.isBlank()) {
        try { return Double.parseDouble(env); } catch (Exception ignore) {}
    }
    String gdk = System.getenv("GDK_SCALE");
    if (gdk != null && !gdk.isBlank()) {
        try { return Double.parseDouble(gdk); } catch (Exception ignore) {}
    }
    String os = System.getProperty("os.name", "").toLowerCase();
    if (os.contains("linux")) return 2.0;
    return 1.0;
}

Ordre de preference :

  1. Variable d'environnement MAGASIN_UI_SCALE (specifique a notre app).
  2. Variable d'environnement GDK_SCALE (standard sur GNOME).
  3. Defaut 2.0 sur Linux (la plupart des laptops modernes).
  4. Defaut 1.0 ailleurs.
